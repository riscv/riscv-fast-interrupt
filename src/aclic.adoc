== Introduction

The set of extensions defined in this specification enhance the existing interrupt specifications.
The Advanced Core Local Interrupt Controller (ACLIC) is a single hart interrupt controller defined for:

  * Low latency interrupt handling
  * High throughput interrupt handling
  * Up to 1023 external interrupts
  * Low area overhead
  * Maximized compatibility with AIA

The table below provides a summary of the ACLIC extensions.

[%autowidth, options="header"]
|===
| Extension Name | Description
2+| Fundamental ACLIC Extensions
| Smidctrl     | Interrupt domain control interface at Machine level
| Ssidctrl     | Interrupt domain control interface at Supervisor level
| Smeihv       | External interrupt hardware vectoring at Machine level
| Sseihv       | External interrupt hardware vectoring at Supervisor level
| Smnip        | Nested Interrupt Preemption support at Machine level
| Ssnip        | Nested Interrupt Preemption support at Supervisor level
2+| Fast Interrupt Extensions for Microcontrollers
| Smijt        | Interrupt jump table at Machine level
| Ssijt        | Interrupt jump table at Supervisor level
| Smehv        | Synchronous exception hardware vectoring at Machine level
| Ssehv        | Synchronous exception hardware vectoring at Supervisor level
| Smcsps       | Conditional stack pointer swap at Machine level
| Sscsps       | Conditional stack pointer swap at Supervisor level
| Smip         | Support for interrupt handler push/pop at Machine level
| Ssip         | Support for interrupt handler push/pop at Supervisor level
|===

This specification aims for single-hart systems and no support for the H extension is foreseen.

NOTE: The extensions defined here are orthogonal to the NMI and RNMI
mechanisms. Their behavior is unchanged by the extensions of ACLIC.

== Fundamental ACLIC Extensions

This section lays the foundation for fast interrupt handling.

To achieve this, it builds upon the hart-level behavior of AIA,
and integrates a subset of Advanced Platform Level Interrupt Controller (APLIC) functionality into a hart-local interrupt controller.

At config-time, the interface differs from AIA,
but the same structures for configuration as in the APLIC are used.

An ACLIC is made up of two major components:

. <<domains>> providing direct delivery of interrupts to the hart (Non-ISA)
. Indirect CSR physical control interface to hart-local APLIC domain registers (<<smidctrl>>)

The latter is additionally made up of

. Interrupt control interface for pending and enable bits based on IMSIC
. New indirect CSRs for accessing the remaining state of the hart-local APLIC domains
. A new CSR {xtopsi} for claiming either major or external interrupts

In addition, extensions are defined for vectoring of external interrupts (<<smeihv>>) and accelerating nested preemption (<<smnip>>).

[[domains, Hart-local APLIC Domains]]
=== Hart-local APLIC domains (Non-ISA)

The APLIC was designed as a multi-hart interrupt controller.
In the context of an ACLIC,
APLIC domains are used tightly coupled with a single hart.
The domains used here are a simplified subset of the full domain definition of the APLIC.

==== Interrupt domain mapping

An APLIC has one or more interrupt domains,
each associated with a subset of RISC-V harts at one privilege level.

An ACLIC system requires at most two domains.
It only requires one domain if M-mode but not S-mode has been implemented,
and two domains if M- and S-mode are implemented.

The root domain is always at machine level.
If supervisor level is implemented,
interrupts can be delegated to a supervisor-level domain,
which is the only child domain (Child Index '0') of the root domain.

==== Simplifications to domain state

The APLIC contains a per-domain configuration register,
which, among other things, allows selecting the delivery mode.

In an ACLIC system, `DM` is fixed to 0 (direct delivery mode).

The following, additional restrictions are applied:

[%autowidth,float="center",align="center",cols="<,<",grid=none,frame=none]
|===
|bits 31:24 |read-only 0x80 
|bit 8|IE = 1 (read-only)
|bit 7|read-only 0
|bit 2 |DM = 0
|===

The `IE` bit is fixed to 1, as its functionality is subsumed by the {xstatus}.`xIE` bits at the hart.
The `BE` bit is now reserved.
Endianness is controlled exclusively by the setting in the {xstatus} register at the hart.

Further simplifications are:

* The Child Index field in the {sourcecfgi} register is always '0' when D=1 (i.e. delegation to child domain). This is only possible in Machine mode. In Supervisor mode, D=0, as there is no child domain.
* As the ACLIC operation is based on direct interrupt delivery, {xmsiaddrcfgh} and {genmsi} registers are not required.
* Interrupt delivery control (IDC) structures are not needed, as the domain has a fixed relationship with one hart, and IDC functionality is already provided by hart-local CSRs.
* The hart index inside the target registers is read-only zero, as only one hart is supported.

APLIC registers or fields not required here shall be read-only zero.

==== Interrupt delivery

The APLIC rules of direct interrupt delivery apply.
External interrupt signals are supplied to the hart from the respective domain.
The interrupts from the root domain appear as MEIP (Machine External Interrupt-Pending),
and interrupts from the child domain appear as SEIP (Supervisor External Interrupt-Pending).

A hart-local APLIC domain signals the priority and identity of the highest priority pending-enabled interrupt to the hart.
This information influences the arbitration between major and minor interrupts that is ultimately reflected in the {xtopsi} CSRs.

[[smidctrl, Smidctrl/Ssidctrl]]
=== Interrupt domain control interface (Smidctrl, Ssidctrl)

Smidctrl depends on the Smcsrind and Smaia extensions.
Ssidctrl depends on the Sscsrind and Ssaia extensions.

NOTE: Many AIA features like programmable IPRIO may not be needed in resource-constrained systems. 
Since these are already controlled by WARL registers, implementations can save area by choosing to not support them. +
 +
A minimal implementation of the AIA `mvip` CSR is suggested by the AIA specification: +
"When supervisor mode is implemented, the minimal required implementation of mvien and mvip has all bits being
read-only zeros except for mvip bits 1 and 9, and sometimes bit 5, each of which is an alias of an existing 
writable bit in mip. When supervisor mode is not implemented, registers mvien and mvip do not exist."
 +
Further, resource-constrained implementations of these extensions are expected to implement the {iprio_lowercase} array of major interrupts as read-only zeros.

The interrupt control interface contains the hart-level interface to operate a hart-local APLIC,
by building on the hart-level features of AIA.

Indirect CSR access to APLIC domains is provided to subsume
the memory-mapped register interface in a hart-local configuration.

This approach has several advantages:

- Built-in privilege level protection: CSRs are clearly mapped to a privilege level.
  Therefore, no additional protection e.g. through PMP is required.
  Especially in systems without memory translation,
  this is beneficial for saving valuable memory protection resources
- Atomic access: Atomic read/write, read & set and read & clear instructions are available for CSRs
- Resource efficiency: Due to smaller access granularity and range,
  implementations using indirect CSRs can be more resource efficient.
- Platform-independent: Memory-mapped base addresses are platform-defined. 
  This places an additional configuration burden in the hands of a user.
  CSR-based interfaces always use the same CSR register numbers

This is achieved by reusing the allocated indirect CSR space for the interrupt pending and enable bits in an IMSIC.
Additional indirect CSR access is provided to the remaining state.

==== Interrupt control interface based on IMSIC interface

ACLIC provides access to interrupt pending and enable bits of the hart-local APLIC domains using the same mechanisms as access to interrupt pending and enable bits in the interrupt files of an IMSIC.

The AIA {eidelivery} and {eithreshold} CSRs defined for the Incoming Message Signaled Interrupt Controller (IMSIC) are not included in the scope of Smidctrl/Ssidctrl.

NOTE: It is not anticipated that ACLIC implementations would allow dynamically switching {eidelivery} into other modes.
If e.g. MSI delivery is required in a system, it is recommended to use the full AIA implementation with IMSIC.
Therefore, no dynamic selection of ACLIC operation via {eidelivery} is provided.

In an ACLIC, {xtopei} is implemented and reports the current highest-priority pending-and-enabled interrupt of the connected APLIC domain.
It reports the minor interrupt identity in {IID}, and the configured priority number in {IPRIO}.

----
xtopei
 XLEN-1:26   (reserved)
 25:16       IID
 15:8        (reserved)
 7:0         IPRIO
----

NOTE: As {eithreshold} is not implemented, any interrupt that has its enable bit set to one is considered enabled for the purpose of reporting it in {xtopei}.

In an ACLIC, registers {eipk} and {eiek} serve the same functionality as with {eidelivery} = 1 in an IMSIC,
i.e. they are the pending and enable bits for an interrupt source `k`.
The `eip` and `eie` arrays act as an alias of the pending and enable bits of the connected APLIC domain.

[NOTE]
====
For an inactive source, the corresponding bits in {eipk} and {eipk} are read-only zero.
In addition, the AIA defined rules "Precise effects on interrupt-pending bits" apply.
====

[NOTE]
====
Providing access to APLIC pending and enable bits at the hart-level has several advantages over using an IMSIC with interrupt delivery from an APLIC

  * It removes unnecessary resource duplication between IMSIC and APLIC when integrated with a single hart.
  * It simplifies the handling of level-sensitive interrupts, avoiding additional checking of the pending flag in the APLIC in a handler of a level sensitive interrupt.
  * It simplifies the register interface, as CSR instructions have single bit operations
====

==== New Top Signed Interrupt Status ({xtopsi}) CSRs

[source]
----
           Number  Name         Description
 (NEW)     0x348   mtopsi       Interrupt top signed interrupt
 (NEW)     0x148   stopsi       Interrupt top signed interrupt
----

The {xtopsi} CSRs are provided for efficiently handling major and external interrupts
in a scheme where interrupts are dispatched by software.

A read of {xtopsi} returns information about the highest-priority pending-and-enabled interrupt for the corresponding privilege level, in this format:

[source]
----
xtopsi
 XLEN-1:16  SIID  Signed Interrupt Identity
 15:8       (reserved)
 7:0        IPRIO
----

If an external interrupt `i` is the highest-priority pending-and-enabled interrupt:

- {SIID} is equivalent to the minor interrupt identity of `i`
- {IPRIO} is the interrupt priority number of `i` configured in the external interrupt controller

Else, if a major interrupt `i` is the highest-priority pending-and-enabled interrupt:

- {SIID} is the two's complement representation of the negated major interrupt identity of `i`
- {IPRIO} is equivalent to what would be reported in {xtopi}.{IPRIO}

In case the {iprio_lowercase} array for major interrupts is read-only zeros,
{xtopsi}.{IPRIO} must be 0 or 255, i.e. the simplified implementation, where {IPRIO} is one when an interrupt is pending, is not allowed.

NOTE: The intention of providing a signed version of interrupt identity is to index into a table with
decreasing memory address per major identity and increasing addresses for minor identities.
A signed table index can be extracted through an arithmetic right shift.

Else, if no interrupt is pending-and-enabled:

- {SIID} is 0
- {IPRIO} is 0

A write to the {xtopsi} CSR claims the reported interrupt identity by clearing its pending bit if possible.
The source value written to {xtopsi} is ignored.

If a read and write of the {xtopsi} CSR are done together by a single CSR instruction (CSRRW, CSRRS, or CSRRC),
the value returned by the read indicates the claimed interrupt.

==== Indirect CSR access to hart-local APLIC domain configuration

To have full control over the necessary APLIC registers, the following indirect CSR access is added.
The registers {xireg4} through {xireg6} are reserved and raise an illegal instruction exception
when {xiselect} is in the range 0x1000-0x10FF.

===== Interrupt Priority (iprio)

When XLEN = 32, each `xireg` register controls the ACLIC priority setting of four interrupts.

[%autowidth]
|===
| `xiselect` |  `xireg` size |  `xireg` state
| 0x1000     |   4B          | `acliciprio0`
| ...        |   ...         | ...
| 0x10FF     |   4B          | `acliciprio255`
|===

Indirect access to `acliciprio[__k__]` mirrors `target[__k*4__].iprio` up to `target[__k*4+3__].iprio`.

When XLEN = 64, only the even-numbered registers exist and each register controls the priority setting of eight interrupts.
Indirect access to `acliciprio[__k__]` mirrors `target[__k*8__].iprio` up to `target[__k*8+7__].iprio`.

===== Source Configuration (sourcecfg)

When accessing the source configuration via indirect CSR, only the lower 16b are accessible,
as the rest of the register is reserved.

When XLEN = 32, the `xireg2` and `xireg3` registers combined control the source configuration of four interrupts.

[%autowidth]
|===
| `xiselect` |  `xireg2/3` size |  `xireg2` state      |  `xireg3` state
| 0x1000     |   4B             | `aclicsourcecfg0`    | `aclicsourcecfg1`
| ...        |   ...            | ...                  | ...
| 0x10FF     |   4B             | `aclicsourcecfg510`  | `aclicsourcecfg511`
|===

Indirect access to `aclicsourcecfg[k]` mirrors `sourcecfg[k*2][15:0]` up to `sourcecfg[k*2+1][15:0]`.
Access to `aclicsourcecfg[0][15:0]` is read-only zero.

When XLEN = 64, only the even-numbered registers exist and the `xireg2` and `xireg3` registers combined control the source configuration of eight interrupts.
Indirect access to `aclicsourcecfg[k]` mirrors `sourcecfg[k*4][15:0]` up to `sourcecfg[k*4+3][15:0]`.

===== State Enable

Additional control is provided for the newly added registers `acliciprio[k]` and `aclicsourcecfg[k]`.

If the Smstateen extension is implemented, then the bit 53 (ACLIC) in {mstateen0} is implemented. If bit 53 (ACLIC) of a controlling {mstateen0} CSR is zero,
then access to the new CSRs (`acliciprio[k]`, `aclicsourcecfg[k]`, {stopsi}) by S-mode or a lower-privilege mode
results in an illegal instruction exception.

If the Smstateen extension is implemented, then bit 58 (IMSIC) in {mstateen0} is implemented, even though IMSIC is not present.
If bit 58 of a controlling {mstateen0} CSR is zero, then access to the CSRs reused from the IMSIC ({eipk} and {eiek})

[[smeihv, Smeihv/Sseihv]]
=== External interrupt hardware vectoring (Smeihv & Sseihv)

In this extension, a new mode is enabled in {xtvec}.{mode}.

[source]
----
 mode  PC on Interrupt
 10    OBASE+4*siid                         # External interrupt vectored mode

where:
  on external interrupt:
    siid = minor interrupt identity
  else, on major interrupt
    siid = - major interrupt identity

  OBASE  = xtvec[XLEN-1:2]<<2   # Vector base is at least 4-byte aligned
----

An example vector table in the new mode could be:

.Hardware vector table layout for `{xtvec}.MODE=10`
[cols="1,1,1",options="header"]
|===
| Event
| Address relative to `OBASE`
| Contents

| custom IRQ N
| `-4 * N`
| j handler_irq_n.

| `...`
| ...
| 

| Machine timer interrupt
| `-4 * 7`
| j interrupt_dispatcher

| Supervisor timer interrupt
| `-4 * 5`
| j interrupt_dispatcher

| Machine software interrupt
| `-4 * 3`
| j interrupt_dispatcher

| Supervisor software interrupt
| `-4 * 1`
| j interrupt_dispatcher

| Synchronous exception
| `+0`
| j exception_handler

| External IRQ 1
| `+4 * 1`
| j handler_ext_1

| External IRQ 2
| `+4 * 2`
| j interrupt_dispatcher

| `...`
| ...
| 

| External IRQ N
| `+4 * N`
| j handler_ext_n

|===

Note, that the table contains some entries, which jump to a common interrupt dispatcher,
and others, which jump to an interrupt source specific handler.

NOTE: When Smeihv and/or Sseihv are supported,
implementation may reserve a sufficient amount (depending on the number of supported interrupt sources) of LSBs as zero in {xtvec},
which avoids a full adder circuit for calculating the program counter value on interrupt trap.

[[smnip, Smnip/Ssnip]]
=== Nested Interrupt Preemption Support (Smnip & Ssnip)

The Smnip extension depends on the Smidctrl extension and adds nested preemption support at machine level.

The Ssnip extension depends on the Ssidctrl extension and adds nested preemption support at supervisor level.

The extensions enable higher-priority interrupts to preempt lower-priority interrupts efficiently in a nested fashion.
It provides additional state to resume execution of the previous handler after a nested preemption.

==== Changed and new CSRs at Machine Level

[source]
----
           Number  Name         Description
 (NEW)     0x346   mistatus     Interrupt status
 (NEW)     0x347   mithreshold  Interrupt enable threshold
 (changed) 0x348   mtopsi       Interrupt top signed interrupt
 (Changed) 0x342   mcause       Cause register
----

Additionally, this extension adds an indirect CSR

[%autowidth]
|===
| {miselect} |  {mireg4} size |  {mireg4} state
| 0x1000     |   XLEN         |  {miconfig}
|===

The registers {mireg}, {mireg2}, {mireg3}, {mireg5}, and {mireg6} are reserved
when {miselect} is 0x1000.

===== New Interrupt Configuration ({miconfig}) CSR

[source]
----
miconfig fields
 bits        description
 XLEN-1:20   (reserved)
 19:16       snipbits (if Ssnip is implemented)
 15:12       (reserved)
 11:8        mnipbits
 7:6         (reserved)
 5           sipu (if Ssip is implemented)
 4           mipu (if Smip is implemented)
 3           snipen (if Ssnip is implemented)
 2           mnipen
 1:0         (reserved)
----

{miconfig} is a WARL register used to configure the behavior of interrupts.

If Ssnip is implemented, the fields {snipbits} and {snipen} of {miconfig}
control nested interrupt preemption behavior for supervisor level.

The supervisor-level CSR {siconfig} is a subset of {miconfig}.
Reading or writing a field of {siconfig} reads or writes the corresponding field of {miconfig}.

The fields {mnipen} and {snipen} enable this extension at M-mode and S-mode, respectively.
Enablement of the extension refers to activating the modified operative behavior specified in <<op-threshold>>.

If Ssnip is not implemented, then {snipbits} and {snipen} are read-only 0.
If Ssip is not implemented, then {sipu} is read-only 0.
If Smip is not implemented, then {mipu} is read-only 0.

The field {miconfig}.{mnipbits}/{snipbits} defines how many most-significant priority bits participate in nested interrupt preemption grouping.
Interrupts whose priorities differ only in less-significant bits cannot preempt each other.
The highest legal value of {miconfig}.{mnipbits}/{snipbits} is IPRIOLEN.
A mask for filtering out the less-significant priority bits that do not participate in nested interrupt preemption at mode `x` is:

[source,subs="attributes"]
----
NIPPRIO_MASK = ~((2^(IPRIOLEN - xiconfig.xnipbits)) - 1)
----

NOTE: For example, if {miconfig}.{mnipbits}=4, 
NIPPRIO_MASK=11110000, i.e. the four MSBs are participating in interrupt preemption.

===== New Interrupt enable threshold ({mithreshold}) CSR

A new WLRL M-mode CSR, {mithreshold},
determines the priority cutoff used to mask lower-priority interrupts (see <<op-threshold>>).

[source]
----
mithreshold
 XLEN-1:9   (reserved)
 8:0        iprio[8:0]  Interrupt-enable threshold
----

The field {mithreshold}.{iprio_lowercase} implements exactly IPRIOLEN+1 bits.

[NOTE]
====
In contrast to existing threshold mechanisms in AIA,
this CSR applies to major and minor interrupt identities.

Because {IPRIO} = 0 is a valid value for major interrupts with a default priority greater
than external interrupts,
{mithreshold}.{iprio_lowercase} = 0 does not enable all interrupts,
and the field needs to be one bit wider than IPRIOLEN in order to allow a threshold
that enables all interrupts.
====

[NOTE]
====
An interrupt threshold is needed in an efficient implementation of synchronization protocols like
Priority Ceiling Protocol (PCP) that are used with preemptive interrupt schemes.
====

===== Changed Top Signed Interrupt Status ({mtopsi}) CSR

The CSR is changed to manage the interrupt threshold when claiming interrupts
to enable efficient nested preemption.

The format of {mtopsi} is changed to accommodate the wider {IPRIO} value needed for nested preemption.

[source]
----
mtopsi
 XLEN-1:16  SIID  Signed Interrupt Identity
 15:9       (reserved)
 8:0        IPRIO
----

The read-behavior is changed in the case that no interrupt is pending-enabled.
In this case

- {SIID} is 0
- {IPRIO} is equivalent to {mithreshold}.{iprio_lowercase}

NOTE: Returning {mithreshold}.{iprio_lowercase} as {IPRIO} field allows software to use this priority as threshold for spurious interrupts handlers.
If it would return zero, software would need special handling of spurious interrupts to prevent writing 0 to the threshold register.

The write-behavior is changed to perform the following stepwise actions:

. The source register bits [16:8] are written to {mithreshold}.{iprio_lowercase}
. The reported interrupt identity is claimed by clearing its pending bit if possible.
. The interrupt threshold is raised by writing the reported {IPRIO} value to {mithreshold}.{iprio_lowercase}

===== Changed Cause ({mcause}) CSR

When {miconfig}.{mnipen} is one,
the exception code in {mcause} is modified as follows:

If a trap was taken on an external interrupt,
the Exception Code field reports the minor interrupt identity.

Else, if a trap was taken on a major interrupt identity,
the Exception Code field reports the two's complement representation of the negated major interrupt identity.

NOTE: The representation of exception code on interrupts is aligned with the {SIID} field in {mtopsi}.

[#op-threshold]
==== Operation

Based on NIPPRIO_MASK derived from {xiconfig}.{xnipbits},
a nested preemption priority threshold is given as

[source]
----
NIP_THRESHOLD = (xithreshold.iprio & NIPPRIO_MASK)
----

When {xiconfig}.{xnipen} is one,
an interrupt source is considered disabled
irrespective of its interrupt-enable bit,
when the following condition holds:

[source]
----
IPRIO >= NIP_THRESHOLD
----

where {IPRIO} is the value reported in the {xtopsi} CSR,
if the interrupt source is the highest priority pending-enabled one.

[NOTE]
====
The highest possible priority value of {IPRIO} is 0.
When {xithreshold}.{iprio_lowercase} and NIPPRIO_MASK is 0,
the nested interrupt preemption threshold is 0 and thus,
all interrupts are considered disabled.
====

[NOTE]
====
The lowest priority value reported by a pending-enabled interrupt in {IPRIO} is 255.
When {xithreshold}.{iprio_lowercase} is 256 or larger,
no interrupts are disabled by the thresholding mechanism.
====

[NOTE]
====
{IPRIO} is derived from the configured priority or default priority order
according to the rules defined in AIA.
====

===== Trap behavior

The existing trap behavior is modified as follows:

When a trap is taken into level `x`,
the current value of {xithreshold}.{iprio_lowercase} is written to {xistatus}.{pithreshprio}.

Additionally, if the trap was taken on an interrupt,
{xithreshold}.{iprio_lowercase} is set to {xtopsi}.{IPRIO}.
This automatically raises the interrupt-enable threshold.
Also, the interrupt is claimed by clearing the pending bit if possible.

If a double trap occurs while in M-mode,
{xistatus}.{pithreshprio} is unchanged.

NOTE: Synchronous exceptions do not modify the value of {xithreshold}.

===== Returns from Handlers

The behavior of an {xret} instruction is modified as follows:

{xret} sets {xithreshold}.{iprio_lowercase} to {xistatus}.{pithreshprio}.

This automatically restores the interrupt-enable threshold to the level of the previous context.

[[smnip-sw-dispatcher]]
===== SW dispatcher

The first example modifies the example handler from AIA by adding support for nested preemption.
When {mtvec}.{mode} = Direct (0), all interrupts use the SW dispatcher as a common entry point.
In vectored mode, some interrupt may jump directly into a custom handler,
if their nested interrupt preemption priority is higher than any interrupt that uses the SW dispatcher.
In this case, the entry in the interrupt table corresponding to those interrupts,
which should use the SW dispatcher,
need to cause the execution to jump to the common SW dispatcher.


[,c]
----

i = read CSR mcause
if (i >= 0) {
    handle synchronous exception i
    restore mstatus if necessary
}
if (mstatus.MPIE == 1) {
    save mistatus
    enable interrupts

    loop {
        call the interrupt handler for major interrupt id

        topsi_val = read/write CSR mtopsi with value of mistatus
        i = (int) topsi_val >> 16
    }

    disable interrupts
    restore mistatus
}
restore processor registers
return from trap
----
(This example can be further optimized, but with an increase in complexity.)

The interrupt threshold is raised automatically on interrupt trap,
and through the write portion of the {mtopsi} access.

If no interrupt is pending-enabled when {mtopsi} is read,
a special handler, located at vector table entry of interrupt id 0,
jumps to the epilogue of the handler.

An example of a preemptible fast interrupt handler is

[,C]

fast_interrupt_handler:
  save mepc, mistatus
  enable interrupts

  /* interrupt handling */

  disable interrupts
  restore mepc, mistatus
  return from trap

==== Example: Threshold behavior under nested traps and interrupts

This example illustrates heavy nesting with explicit threshold management,
under the following assumptions:

* Any handler must save xistatus before enabling interrupts
* Any handler that had interrupts enabled at any time must restore xistatus with interrupts disabled before executing xret

.Preconditions
* Thread **T** starts with `xithreshold = 128`.
* Interrupt **A** has `IPRIO = 32`; interrupt **B** has `IPRIO = 4`; interrupt **C** has `IPRIO = 2`.

| 0  | T (thread) | —                         | 128 | X   | Baseline: thread running with cutoff 128.

| 1  | A          | save context; IE=1        | 32  | 128 | Save xistatus and enable interrupts afterwards.
| 2  | A          | raise threshold           | 2   | 128 | Short critical section in A; `pithreshprio` unchanged.
| 3  | A          | lower threshold           | 32  | 128 | End of A's critical section; restore handler threshold to 32.
| 4  | B          | trap (interrupt B)        | 4   | 32  | Trap writes `pithreshprio=32`; interrupt entry sets `xithreshold=4`.
| 1  | B          | save context; IE=1        | 4   | 32  | Save xistatus and enable interrupts afterwards.
| 5  | E          | trap (exception E)        | 4   | 4   | Exception trap captures current threshold into `pithreshprio=4`.
| 1  | E          | save context; IE=1        | 4   | 32  | Save xistatus and enable interrupts afterwards.
| 7  | C          | trap (interrupt C)        | 2   | 4   | Trap writes `pithreshprio=4`; interrupt entry sets `xithreshold=2`.
| 9  | C→E        | return (xRET)             | 4   | 4   | On C's return, `xithreshold` restored from `pithreshprio=4`; resume E.
| 1  | E          | IE=0; restore context     | 4   | 4   | Disable interrupts and restore xistatus afterwards.
| 10 | E→B        | return (xRET)             | 4   | 4   | E returns; `xithreshold` restored from `pithreshprio=4`; resume B.
| 1  | B          | IE=0; restore context     | 4   | 32  | Disable interrupts and restore xistatus afterwards.
| 11 | B→A        | return (xRET)             | 32  | 32  | On B's return, `xithreshold=32` from `pithreshprio=32`; resume A.
| 1  | A          | IE=0; restore context     | 32  | 128 | Disable interrupts and restore xistatus afterwards.

== Fast Interrupt Extensions for Microcontrollers

This section introduces additional extensions,
which can reduce latencies for fast interrupt handling.

The features specified here are each capable of shaving of several cycles of a worst-case interrupt latency.
They include accelerating a conditional swap of the current stack pointer with a more privileged one (<<smcsps>>),
HW-vectoring of interrupts using interrupt jump tables (<<smijt>>), and HW-vectoring synchronous exceptions (<<smehv>>),
and HW-assisted save and restore (<<smip>>).

NOTE: These extensions are targeted at microcontrollers with a comparatively simple microarchitecture.
There is an implementation-dependent cut-off,
at which the benefit of these extensions does not justify the additional complexity imposed on the microarchitecture.
This is expected to be the case for deeply pipelined machines.

[[smcsps, Smcsps/Sscsps]]
=== Conditional Stack Pointer Swap extension (Smcsps, Sscsps)

The Smcsps extension adds abilities for efficient conditional stack pointer swap at machine level.
The Sscsps depends on the Smcsps extension and adds abilities for efficient conditional stack pointer swap at supervisor level.

For security and stack safety reasons, it can be necessary to separate the stack areas per privilege level and between traps and normal execution.
Performing stack pointer swapping in SW can be time consuming, as the conditions for swapping have to be checked first.
However, stack pointer swapping is among the first things that need to happen even before saving the interrupted context.
To ensure fast interrupt handling, this extension introduces instructions to accelerate conditional stack pointer swapping.

==== New stack pointer for conditional swap ({xspcs}) CSR

New CSRs are introduced to hold an alternative stack pointer.
Conditionally, this stack pointer can be swapped with the currently active one.

[source]
----
      Number  Name         Description
(NEW) 0x349   mspcs        M-mode stack pointer for conditional swap (WARL)
(NEW) 0x149   sspcs        S-mode stack pointer for conditional swap (WARL)
----

Each CSR holds a full XLEN-wide value, that can be swapped with the designated stack pointer register (x2).

NOTE: As per the RISC-V psABI, stack pointers are at least 4 B aligned.
Therefore, at least the two least significant bits of the stack pointer are zero in psABI compliant code.
Nevertheless, a full XLEN-wide register is provided,
in case a trap happens during non-psABI compliant code.
It is the responsibility of the programmer
to write a psABI compliant stack pointer value into {xspcs}
before a stack pointer swap can occur.

==== Instructions for conditional stack pointer swap

To simplify and accelerate the conditional stack pointer swap, two instructions per mode are introduced:
<<#insns-mcspspush>>, <<#insns-mcspspop>>, <<#insns-scspspush>>, and <<#insns-scspspop>>.
To decide if a stack pointer swap is required, these instructions evaluate dedicated status bits.
The {xistatus}.{sppush} (Stack pointer push) bit indicates that the stack pointer in {xspcs} shall be swapped during the push sequence of the next handler.

The {xistatus}.{psppush} (Previous stack pointer push) bit indicates that the stack pointer shall be swapped back into {xspcs} during the pop sequence.
It corresponds to the value of the {xistatus}.{sppush} bit in the previous context.
This bit is part of the context of an interrupt handler, and needs to be saved and restored if a handler can be preempted.

Software can deactivate the stack pointer swap mechanism by writing zero to {xistatus}.{sppush}.

An example sequence with one trap from S- to M-mode and one preemption within M-mode will look like this:

[source]
----
Mode  Event            sppush  psppush  sp-swap
S     -                1       X        n
M     trap             1       X        n
M     mcspspush        0       1        y
M     save context     0       1        n
M     trap             0       1        n
M     mcspspush        0       0        n
M     mcspspop         0       0        n
M     mret             0       0        n
M     restore context  0       1        n
M     mcspspop         1       0        y
S     mret             1       0        n
----

In this example,
the stack pointer is swapped when executing the mcspspush instruction
after taking the trap from S-mode into M-mode,
and is swapped back through mcspspop before returning to S-mode.
No change of stack pointer occurs for the mcspspush and mcspspop instructions in the nested handler.

==== State Enable

If the Smstateen extension is implemented,
then the bit 53 (ACLIC) in {mstateen0} is implemented.
If bit 53 (ACLIC) of a controlling {mstateen0} CSR is zero,
then access to the new CSR {sspcs} by S-mode or a lower-privilege mode
results in an illegal instruction exception.

[[smijt, Smijt/Ssijt]]
=== Support for interrupt jump table (Smijt, Ssijt)

These extensions add a new mode for taking interrupts via a dedicated interrupt jump table.
In contrast to other interrupt vector modes,
an interrupt jump table (IJT) is used to perform hardware vectoring of interrupts.

In this new mode, when an interrupt is taken, the hart hardware loads the interrupt jump
table  (table pointed to the new {xijt} CSR) entry for the associated interrupt,
masks off the least-significant bit (for IALIGN=16) or masks off the two least-significant bits (for IALIGN=32),
and then jumps to the masked address.

This increases the range of reachable addresses to the entire address space.
This is useful when handlers are more than 2 MiB away from the vector table entry.

==== Changed and new CSRs

[source]
----
       Number  Name         Description
       0x305   mtvec        M-mode Trap-handler base address / interrupt mode
 (NEW) 0x307   mijt         M-mode Interrupt jump table base address
       0x105   stvec        S-mode Trap-handler base address / interrupt mode
 (NEW) 0x107   sijt         S-mode Interrupt jump table base address
----

A new base address for the interrupt jump table is introduced.

The WARL XLEN-bit {xijt} CSR
holds the base address of the interrupt jump table.
The address held in {xijt} is aligned on a 64-byte or greater power-of-two boundary.

===== New {xtvec} CSR Mode

Interrupt jump table mode is encoded as a new state in the
existing {xtvec} WARL register, where {xtvec}.{mode} (the two
least-significant bits) is `11`.

The PC upon interrupt is changed as follows:

[source]
----
 mode  PC on Interrupt
 11    handler_addr   # Interrupt jump table mode

where:

  M[a]   = Contents of memory address at address "a"
  entry = M[JTBASE+(vtoffset << (xijt.SHAMT + 2))]
  VEN    = entry & 0x1  # Vectoring enable
  JTMASK = ~0x1 if IALIGN=16 or ~0x3 if IALIGN=32  # Jump table mask
  JTBASE = xijt[XLEN-1:6]<<6  # IJT base is at least 64-byte aligned

  if external interrupt:
    vtoffset = Minor Interrupt Identity
  else
    vtoffset = - (Major Interrupt Identity)

  if VEN = 1
    handler_addr = entry & JTMASK
  else
    handler_addr = xtvec[XLEN-1:2]<<2
----

In interrupt jump table mode, when interrupts are taken, the interrupt behavior is modified as follows:
After executing the required side-effects as required with the existing behavior,
the hart then reads an XLEN-bit handler address with permissions corresponding to the handler's mode
from the in-memory table whose base address (JTBASE).
The trap handler function address is read from `JTBASE + (vtoffset << ({xijt}.{SHAMT} + 2))`.

If the read is successful and the LSB of the entry is 0,
the interrupt should not be hardware-vectored.
Instead, the trap is redirected to a common entry point defined by {mtvec}.

NOTE: Even with hardware vectoring, some interrupts are expected to use a common SW dispatcher.
By deasserting the LSB of their interrupt jump table entry,
a trap will take them to a SW dispatcher at a common entry point indicated by {mtvec}.
This SW dispatcher may access the table pointed to by {xijt} to retrieve the interrupt handler address.

If the read is successful and the LSB of entry is one,
the hart clears the low bit(s) (depending on IALIGN) of the handler address
and sets the PC to this handler address.

NOTE: The original vectored mode simply jumps to an address in
the trap vector table, while the interrupt jump table mode reads a
handler function address from the table, and jumps to it in hardware.

NOTE: In case of a level-sensitive interrupt,
it might not be possible to clear the interrupt-pending bit.

For permissions-checking purposes, the memory access to retrieve the
function pointer for hardware vectoring is a read at the
privilege mode of the interrupt handler, and requires read permission.

It is recommended that the interrupt jump table reads are ignored for hardware triggers and breakpoints.

NOTE: The "jump via table" instructions of Zcmt have set a precedent for treating indirect jumps via tables as fetches.
However, treating the table access as a read is better aligned with existing PMP and ePMP capabilities.
E.g., it allows sharing an interrupt jump table,
where M-mode has read/write permissions and S-mode has read-only permissions.
Such a mode is useful, if interrupts can be installed at run-time.
Also, treating these accesses as reads harmonizes the permissions with accesses into the table
by a SW dispatcher based on interrupt identity.

NOTE: If {xijt} is configured such that multiple privilege levels
share a single interrupt jump table, the memory protection units
(e.g., PMP, Smepmp) must grant read permission to the shared interrupt jump table
for all applicable lower privilege levels. +
 +
Example 1: Standard PMP +
For a system with M-mode and S-mode, a PMP entry with L=1, R=1, W=0 and X=0
permits both privilege levels to execute from the table while preventing modifications.
If L=0, S-mode retains execute only permission while M-mode maintains full permissions. +
 +
Example 2: Smepmp Extension +
Under the Smepmp extension, the shared interrupt jump table can be secured as a "Shared data region".
Setting L=0, R=0, W=1 and X=0 grants both M-mode and S-mode read only permission to the interrupt jump table.
Additionally, M-mode has write permissions to the interrupt jump table.
If S-mode requires modifying the table,
this requires either a system call or trap-and-emulate operation.

It is recommended that the handler fetch be ignored for hardware triggers and breakpoints.

[[interrupt-fault-handling, Interrupt Fault Handling]]
==== Interrupt Fault Handling

This extension introduces side effects to an interrupt trap,
which can cause faults.
With this extension,
faults can occur during the interrupt jump table read.
Therefore, this section defines the required behavior
if a fault occurs while taking an interrupt trap,
before executing the first instruction of the handler.

NOTE: Faults that can occur include __Load page fault__ and __Load access fault__.

If a fault occurs while taking an interrupt trap,
before executing the first instruction of the handler,
the interrupt trap must be aborted,
and a synchronous exception trap must be taken instead.
Such a synchronous exception trap following an aborted interrupt trap is referred to as a precise interrupt fault trap.

Precise interrupt fault traps require that:

* the instruction at the `pc` where the aborted interrupt trap was taken is considered the trapping instruction;
* all instructions older than the trapping instruction have committed their results;
* no instructions newer than the trapping instruction have altered architectural state;
* no operations within the aborted interrupt trap have altered architectural state,
  unless the altered state allows restarting and completing the interrupt trap while still producing the correct final architectural state.

The last requirement is relaxed to allow speculative modification of parts of the CSRs,
specifically those that are overwritten by a trap.
This includes, for example, {xcause} and fields of {xistatus},
including {xpp}, {xpie}, and {pithreshprio},
as these fields will be overwritten by the synchronous exception trap.

NOTE: For simpler systems, {xtval} is not required to be written with the faulting address.
For systems that need to recover from a page fault,
{xtval} should be written with the faulting address to enable page-fault handling code.
Because the instruction pointed to by {xepc} is not the instruction that caused the fault,
it should not be considered during fault handling.
Instead, fault handling for aborted interrupt traps should use only {xcause} and {xtval}.

==== Handler support

The extension is expected to be used in conjunction with the <<smnip>> extensions.
In this case, the handlers introduced in Sections <<smnip-sw-dispatcher>> and <<smnip-fast-handler>> work unmodified.

For the SW dispatcher, an additional configuration assumption is
that interrupts trapping to the SW dispatcher have the LSB of their interrupt jump table entry zero.
Interrupts using a fast handler must have the LSB of their interrupt jump table entry set to one.

==== State Enable

If the Smstateen extension is implemented,
then the bit 53 (ACLIC) in {mstateen0} is implemented.
If bit 53 (ACLIC) of a controlling {mstateen0} CSR is zero,
then access to the new CSR {sijt} by S-mode or a lower-privilege mode
results in an illegal instruction exception.

Accelerating synchronous exception handlers through vectoring may reduce interrupt latency.
During synchronous exceptions, interrupts are disabled.

[[smip, Smip/Ssip]]
=== Interrupt handler push and pop extension (Smip & Ssip)

The Smip and Ssip extensions depend upon the Smcsps and Sscsps extensions, respectively.

The interrupt handler push and pop extension adds automatic (partial) context save and restore behavior.
Interrupt handler latency is reduced by allowing the context save to happen in parallel with the handler fetch,
which may include the read of the interrupt jump table entry with Smijt/Ssijt.

NOTE: In Harvard architectures, the data bus may be idle while the vector table read and handler fetch are happening.
By starting the context save in parallel, the available data bus bandwidth can be utilized, reducing interrupt latency.

NOTE: In some implementations, executing the context save in parallel to the read of the table entry and the first instruction
may contribute to reducing the jitter in interrupt latency,
by masking variable delays e.g. caused by cache misses on handler fetch.

NOTE: Additional latency reduction can be achieved by implementing late-preemption of the push sequence in favor of a higher-priority interrupt or synchronous exception.
Interrupt handling throughput is increased by allowing tail-chaining of interrupts or synchronous exceptions, skipping the context restore and save in between servicing two handlers.
This is not an architectural requirement, but a possible implementation choice.

This extension does not affect Non-Maskable Interrupts (NMIs).

==== Controlling the extension behavior

This extension adds additional control bits to enable/disable the extended behavior.

If {xiconfig}.{xipu} is set, a trap taken at privilege level `x` due to an interrupt causes automatic stacking.

If the bit is clear, the trap behavior at the respective level is unmodified.

NOTE: The state of these bits does not influence the behavior of {mipopret}/{sipopret}.

==== Interrupt trap behavior

The existing trap behavior is modified.
Before executing the first instruction of the trap handler,
this extension partially save the interrupted context.

It performs the following operations in order:

  * Conditionally performs a stack pointer swap by executing `{xcspspush} sp, sp`
  * Save registers x10-x15 (these are the caller saved registers available in RVC)
  * Adjusts the stack pointer to create the stack frame

NOTE: The stack pointer swap is performed as the first action,
as the integrity of the sp of a less privileged level cannot be guaranteed at all times.

The size of the stack frame created is set to XLEN bytes,
which is the minimum stack frame size that can hold the six pushed registers,
and is psABI-compliant.

NOTE: It is the responsibility of the programmer to ensure
that sp contains a psABI-compliant stack pointer value
before the stack pointer is adjusted.
If this is not compliant before that,
the value of sp after the push sequence is also not compliant.

NOTE: As this extension modifies interrupt trap behavior but not the behavior of synchronous exception traps,
it should only be used in interrupt handling modes that provide separate entry points for
interrupt and synchronous exception traps.
Specifically, mtvec.mode = 0 (direct) is incompatible.

NOTE: This extension does not require a normative stack layout.
Interrupt handlers do not require access to the interrupted context.

==== Returns from interrupt handlers

When executing an {xipopret} instruction, the context saved during the trap is restored and a return is executed.

NOTE: An explicit interrupt return instruction is included, rather than modifying the xret behavior,
so that the hart does not need to do bookkeeping whether it is in an interrupt or exception context.

It performs the following actions in order:

  * Adjusts the stack pointer to destroy the stack frame
  * Restores registers x10-x15
  * Conditionally performs a stack pointer swap by executing `{xcspspop} sp, sp`
  * Executes an {xret} according to the current level

NOTE: As this instruction is intended to be executed at the end of the handler, interrupts are typically disabled during its execution.
It is however an implementation choice to perform tail-chaining to the next highest priority pending-enabled interrupt upon this instruction.

==== C interrupt handlers

When this extension is implemented, a GNU compiler may assume for any interrupt handler defined with the interrupt attribute like

[source,c]
----
void interrupt_handler() __attribute__((interrupt("push-pop")));
----

that the registers x10-x15 have been saved.
Additionally, the compiler does not need to restore these registers.
Finally, it should generate an {xipopret} instruction at the end of the function instead of an {xret}.

The attribute can be combined with other attributes, e.g. like

[source,c]
----
void interrupt_handler() __attribute__((interrupt("preemptible", "push-pop")));
----

To indicate that the nested preemption is used in addition.

==== Late Preemption of the interrupt handler

While executing the trap side effects of an interrupt, it is allowed for an implementation to optionally check for higher priority pending and enabled interrupts at the current privilege mode, and switch to fetching this interrupt handler instead.

==== Tail chaining of interrupt handlers

While executing the {xipopret} instruction,
it is allowed for an implementation of a hart
to optionally check for any pending and enabled interrupts at the current privilege mode.

If such an interrupt exists, the hart may abort executing the {xipopret} instruction and trigger the following actions:

  - Update the {xcause}.`Exception Code` to the corresponding interrupt id.
  - Perform the interrupt handler fetch.
  - Start executing the interrupt handler, while skipping the portions of context restore that have not yet been executed by the aborted {xipopret}.

Explicitly, {xepc}, {xistatus} remain unchanged.

NOTE: In the typical case of tail chaining, the check for another pending enabled interrupt is at the start of the {xipopret} instruction.
In this case, the entire context save and restore sequence can be skipped.

==== Interrupt Fault Handling

This extension introduces additional operations during an interrupt trap
that can cause faults.
Specifically, the store operations can cause faults such as
__Store/AMO access fault__ or __Store/AMO page fault__.

For the purpose of handling such faults,
the interrupt trap is aborted and a precise interrupt fault trap is taken as a synchronous exception,
as described in <<interrupt-fault-handling>>.

In idempotent memory regions,
the store operations of the aborted interrupt trap may have updated memory.
Non-idempotent memory regions must not have been updated during the aborted interrupt trap.

  lr a2, OFFSET(sp)         # Restore a2
  lr a3, OFFSET(sp)         # Restore a3
  lr a4, OFFSET(sp)         # Restore a4

  addi sp, sp, FRAMESIZE    # Adjust stack pointer

where
  lr is lw on RV32 and ld on RV64

The order and access size of the loads, as well as the OFFSETS, are implementation-defined.

===== sipopret

Synopsis::
Pop interrupt context and return

Mnemonic::
sipopret

Encoding::
[wavedrom, ,svg]
....
{reg: [
  {bits: 7, name: 0x73, attr: ['SYSTEM'], type: 8},
  {bits: 5, name: 0x0, attr: [''], type: 2},
  {bits: 3, name: 0x0, attr: ['PRIV'], type: 8},
  {bits: 5, name: 0x0, attr: [''], type: 4},
  {bits: 5, name: 0x8, attr: [''], type: 8},
  {bits: 7, name: 0x8, attr: [''], type: 8},
]}
....

Description::
Restores an interrupted context at the end of an interrupt and returns to the previous context

Behavior::

Can be executed at machine or supervisor level.
Attempting to execute in a mode less privileged than supervisor level
will raise an illegal-instruction exception.
The behavior in RISC-V assembly is as follows:

[source]
----
sipopret:
  lr a0, OFFSET(sp)         # Restore a0
  lr a1, OFFSET(sp)         # Restore a1
  lr a2, OFFSET(sp)         # Restore a2
  lr a3, OFFSET(sp)         # Restore a3
  lr a4, OFFSET(sp)         # Restore a4
  lr a5, OFFSET(sp)         # Restore a5
  addi sp, sp, FRAMESIZE    # Adjust stack pointer
  scspspop sp, sp           # Swap stack pointer with sspcs
  sret

where
  lr is lw on RV32 and ld on RV64
----

The order and access size of the loads, as well as the OFFSETS, are implementation-defined.

Included in: <<smip,Ssip>>
